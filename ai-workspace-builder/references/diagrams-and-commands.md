# Diagrams & Commands Guide

## DIAGRAMS — 8 Universal + 3 Code Graphs

Every diagram is mermaid format. Colors use hex fills. Keep under 40 lines.
Claude reads these INSTEAD of text explanations — 300 tokens vs 1500.

### context-map.mermaid (MOST IMPORTANT)
Type: `graph TD`
Purpose: "What to read per task type" — Claude reads this to decide which files to load.
Structure:
```
Session Start → STATE.md → Task type?
  → Write code → [specs, diagrams, agents, rules, stacks]
  → Review code → [code-review skill, coding rules, testing rules]
  → Plan → [orchestrator, project-manager, workflow diagram]
  → Design UI → [ux-designer, frontend-dev, design rules]
  → Fix bug → [qa-engineer, testing rules]
  → Launch → [growth-marketer, devops, monetization diagram]
All paths → Update STATE.md
```

### agent-routing.mermaid
Type: `graph LR`
Purpose: Who handles what. Shows orchestrator → department groups → individual agents.

### workflow.mermaid
Type: `graph TD`
Purpose: Full dev flow: Idea → Spec → Diagram → Plan → Build → Review → Test → Deploy → Measure → Scale/Pivot

### decision-tree.mermaid
Type: `graph TD`
Purpose: "What pattern to use?" Matches refactoring.guru patterns.
Structure: Need? → Create/Structure/Coordinate → specific pattern

### blast-radius.mermaid
Type: `graph TD`
Purpose: Change analysis model: Changed node → direct dependents → indirect → tests → types

### agents-communication.mermaid
Type: `graph LR`
Purpose: Who reports to whom, what flows between agents.

### monetization.mermaid
Type: `graph TD`
Purpose: Funnel: Discover → README → Install → First use → Regular use → Hit limit → Pay → Recommend

### session-lifecycle.mermaid
Type: `graph LR`
Purpose: /start → Read STATE → Work → /end → Save STATE → Next session

### Code graph diagrams (generated by /build-graph, not pre-created)

## COMMANDS — 12 Slash Commands

### Command Format
```markdown
---
name: [command-name]
description: "[what it does]. Usage: /command-name [args]"
---
[Numbered steps Claude follows when this command is invoked]
[Reference to files Claude must read]
[Output format]
```

### Command Catalog

**Session management:**
- `/start` — Read STATE.md + last CHANGELOG entry. Output 5 lines. Don't read anything else.
- `/end` — Update STATE.md + CHANGELOG.md + DECISIONS.md. Output 2 lines.

**Agent activation:**
- `/activate [agent]` — Read STATE.md + agents/[name].md + relevant rules. Follow that agent's checklist.

**Stack setup:**
- `/setup-stack [description]` — Generate stacks/active.md with real code patterns. Save template copy. Create stack-specific skills/agents only if needed. Update STATE.md. Never touch universal config.

**Planning:**
- `/plan` — Structured plan for single task (from skills/plan.md format)
- `/plan-sprint` — Sprint plan with gantt timeline + agent assignments + parallel lanes
- `/create-spec [name]` — Create specs/[name]-spec.md + diagrams/[name].mermaid

**Implementation:**
- `/implement [spec]` — BEFORE: read graphs + spec + diagram. DURING: follow rules + stack. AFTER: update graphs + verify blast radius. ALWAYS: update STATE.md.

**Review:**
- `/review [file]` — Standard code review against rules/coding.md + rules/testing.md
- `/review-impact [file]` — Code review + blast radius from code graph diagrams. Read diagrams first, then only impacted files.

**Code graphs (diagram-based, no MCP):**
- `/build-graph` — Read source signatures. Generate 3 mermaid diagrams (code-graph, code-deps, code-tests).
- `/blast-radius [target]` — Read the 3 graph diagrams. Trace 2 hops of dependents. Output: direct/indirect impact + tests + risk level.

### Command Design Principles
- Every command starts by reading STATE.md (except /start which IS the STATE reader)
- Every command that changes code ends by suggesting /build-graph to update graphs
- Every implementation/review command references rules/coding.md
- Commands use file references ("Read specs/$ARGUMENTS-spec.md"), never inline content
- No command should require >500 tokens of instruction text
